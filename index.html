Key Features

Everything you need, nothing you don't

A production-ready CMS platform with a clean architecture foundation — extensible, secure, and headless by design.

Schema Builder

Visual drag-and-drop field builder with 30+ field types. Add, remove, and reorder fields without any schema migrations. JSON options per field type for fine-grained control.

EAV Content

Entity-Attribute-Value storage for fully dynamic content. Every content item stores typed field values without requiring database schema changes per content type.

Media Library

Upload, crop, and search images with a searchable media picker. Cropper.js in-browser crop, SixLabors.ImageSharp server-side processing. Gallery field for multi-image sets.

Multi-Language

Per-tenant language configuration. Per-field localization flags. Language switcher tabs in the content editor. DB-backed admin UI i18n managed globally by SystemAdmin.

Users & Permissions

Four roles: SystemAdmin, TenantAdmin, Editor, Viewer. Per-content-type CRUD permission matrix. Action filters enforce server-side; UI hides unauthorized operations.

Audit Log

Dual-channel logging: Serilog diagnostic sink (SQL Server / Graylog) + Dapper audit trail. Every write operation recorded with user, tenant, action, and entity info.

SystemAdmin Console

Cross-tenant platform console at /system. Tenant CRUD with atomic provisioning, impersonation, and global UI translation management.

Email 2FA

Optional email-based two-factor authentication via SMTP. Configurable per deployment — EmailVerification:Enabled flag in appsettings. BCrypt workfactor 12.

Docker Deploy

Multi-stage Dockerfile and docker-compose.yml included. SQL Server 2022 runs in a named-volume container. Numbered SQL migration scripts applied via sqlcmd.

EAV Content Structure

Schema-free content, zero migrations

The Entity-Attribute-Value model lets you define any content structure without touching the database schema. Add fields, reorder them, and localize each one independently.

content_types
└─ "Blog Post" CT
├─ content_fields
├─ title Text
├─ body RichText
├─ cover_image Image
├─ category Relation
├─ authors MultiRelation
└─ is_featured Boolean
└─ content_items
├─ content_field_values
└─ content_field_relations
Example: Blog Post fields
titleTextlocalized
bodyRichTextlocalized
cover_imageImageglobal
categoryRelationglobal
authorsMultiRelationglobal
is_featuredBooleanglobal
→ Add fields without schema changes
New field = one row in content_fields. Values land in content_field_values (typed columns: value_text, value_number, value_bool, value_date, value_media_id). No ALTER TABLE ever.
→ Relation fields use a dedicated junction
content_field_relations stores Relation and MultiRelation values — ordered by sort_order, expanded to {id, displayValue} in API responses.
→ Per-field language isolation
is_localized = 1: value stored per language_code. is_localized = 0: stored once with language_code = 'all'. Content editor shows per-language tabs for localized fields.
→ Gallery field = one row, JSON array
Gallery stores an ordered JSON array of media IDs in value_text. The UNIQUE(item, field, lang) constraint enforces single-row semantics. API expands to [{id, url}, ...].
Public REST API

API Management

Expose any content type as a typed REST endpoint in one click. Authentication, rate limiting, field visibility, aliasing, caching — all configurable per content type from the admin panel.

camelCase Field Key Convention

All fields keys in every response are camelCase. Aliases are entered as kebab-case in admin — the system normalizes automatically.

ExternalKey = ToCamelCase(alias ?? field.Slug)
slug: "seo-meta-description" → seoMetaDescription
alias: "reading-time" → readingTime
slug: "published-at" → publishedAt
slug: "hero-title" → heroTitle

Same keys apply for filtering (filter[readingTime]=5), sorting (sort=publishedAt:desc), and projection (?fields=heroTitle).

bash — curl example 
# GET with lang, sort, and API Key auth
$ curl \
  "https://dev-tenant.cms.example.com/api/v1/
dev-tenant/blog-post?lang=en&sort=publishedAt:desc&page=1" \
  -H "X-API-Key: vk_1_demo123secret"

# With field projection (camelCase keys)
$ curl \
  "...?fields=title,seoMetaDescription,publishedAt" \
  -H "X-API-Key: vk_1_demo123secret"

# Filter by camelCase field slug
$ curl \
  "...?filter[isFeatures]=true&filter[readingTime]=5" \
  -H "Authorization: Bearer eyJ..."

lang=en
Language code for localized fields. Defaults to tenant's default language.
page=1&pageSize=20
Pagination. Requires allow_pagination = 1 on the API config.
filter[fieldKey]=value
EAV field filtering. Key is camelCase slug or alias. Requires allow_filtering = 1.
sort=publishedAt:desc
Sort by camelCase field key or built-ins: id, createdAt, updatedAt, status.
fields=title,category
Field projection — camelCase keys only. Reduces response payload.
X-RateLimit-* headers
Sliding-window per tenant+config. Default 60 req/min. HTTP 429 + Retry-After on exceed.

Quick Start

Up and running in minutes

Choose Docker for the quickest path — SQL Server and migrations handled in three commands. Or use the classic publish path for production environments.

A Docker
bash 
# Step 1 — Start SQL Server container
$ docker-compose -f docker/docker-compose.yml up -d

# Step 2 — Apply schema migration
$ docker exec varyo_db \
  /opt/mssql-tools18/bin/sqlcmd \
  -S localhost -U sa -P "YourPass!" -C \
  -d Varyo -i /migrations/001_v1_schema.sql -b

# Step 3 — Apply dev seed (optional)
$ docker exec varyo_db \
  /opt/mssql-tools18/bin/sqlcmd \
  -S localhost -U sa -P "YourPass!" -C \
  -d Varyo -i /migrations/002_v1_dev_seed.sql -b

# Step 4 — Run the app
$ dotnet run --project src/VaryoCms.Web

info: Now listening on http://localhost:5267
# Login: admin@dev.local / Admin123!
B Classic Publish
bash 
# Step 1 — Apply migration via sqlcmd
$ sqlcmd -S your-server -U sa \
  -P "YourPass!" -d Varyo \
  -i db/migrations/001_v1_schema.sql -b

# Step 2 — Publish the app
$ dotnet publish src/VaryoCms.Web \
  -c Release -o ./publish

# Step 3 — Set environment variables
$ export ConnectionStrings__DefaultConnection=\
  "Server=...;Database=Varyo;..."
$ export Jwt__SigningKey="your-256-bit-secret"
$ export DevTenantSlug="your-tenant"

# Step 4 — Run
$ dotnet publish/VaryoCms.Web.dll

info: Now listening on http://0.0.0.0:5267
